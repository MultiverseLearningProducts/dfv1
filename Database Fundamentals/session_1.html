Relational Databases
There are several types of database that organisations use, but one of the most common is the relational database which we will be focussing on today. We will see some of the others towards the end of the session.
Relational Database Manaagement System (RDBMS):
A database where data is organised into tables with defined relationships between them
A RDBMS is your typical database structure where data is stored across various tables depending on their use. For example, when selling a product you may have a table showing sales with another holding product information. There will be a defined relationship between the two by both having a common field. Most of the data you use will likely be stored in a RDBMS. Let's look at some typical characteristics.

If all your data is one table then updating becomes a chore. Imagine a table where each row was a family member and included their house number. Keys
We have discussed what a RDBMS is and that for the tables stored within there are defined relationships between them. But how are these relationships defined, the 'key' is keys...

Primary Key

A unique identifier for each row in a table

coach_table
Coach Id Coach Name No. Cohorts Assistant
1 Steph 5 Ashray
2 Ben 2 Ashray
Keys are how relationships are defined between tables and there exists a few different types. The first is the primary key which is a unique identifier for each row in a table such that you can access any record by referencing this value. A key cannot be repeated and must be unique for each record. Often this will be the first column and will be an integer (but not always...). Every table in a RDBMS must have a primary key.

Primary keys...

...cannot be NULL

...must be unique

...should rarely be changed

...given a new value when a new record is created

A few rules about primary keys, they must hold a value (not null), each value must be unique to each record, they should rarely be changed and whenever a new record is added a key should be assigned.

Foreign Key

A field in one table that is a primary key in another table . These keys enable relationships between tables and allow them to be joined.

A foreign key is a field that points to a primary key in another table. In this example each apprentice is assigned a coach who is identified by their ID. This value is the primary key for the coach in the coach_table. The foreign key establishes the relationship between the two tables.

Composite Key

Two or more columns together acting as a primary key

assignment_table
apprentice_id module_id grade coach_id
1 1 99 1
1 2 75 1
2 1 80 2
2 2 78 2
What happens though if it is impossible to give each row in a table a unique identifier? For example, when scanning items through a checkout at a supermarket, each item will require its own row in a table (to record price and other product info) but will be part of the same transaction, so will have the same ID. This is where a composite key is used. When two fields together uniquely identify a record, you have a composite key. In this example, each apprentice and module will appear several times, but the combination of each will be unique to each row.

Partial vs Transitive Dependency

A common source of confusion is understanding the difference between transitive and partial dependency, let's look at both.

Partial dependency occurs when a table has a composite key and a field is dependent on one part of it

Transitive Dependency occurs when a field can be inferred from another field that is not the primary key

Partial dependency occurs in tables with composite keys where some of the fields are only dependent on part of the key. For example, the city field only relates to the vendor and not the item_id as that value will appear if the product is sold in another city. That is a partial dependency. A transitive dependency occurs when you cna understand a field from another which isn't the primary key. I know Essex has a large population because I can see that from the number.

NoSQL
Before we finish for the day, let's discuss another common database type- Not Only SQL (or NoSQL).
NoSQL ("Not only SQL") is an alternative to traditional relational databases that can accomodate a wide variety of data models.
A RDBMS is well suited for structured data in tabular formats, but as we know from the previous module, not all data is structured. You can have emails, videos, audio files, reports and more which all come under the unstructured umbrella. NoSQL databases provide a way of retrieving this sort of data.

Document Orientated Database

A common NoSQL database where all instances of an object is stored in one document as opposed to spread across multiple tables. To access the data you reference the internal structure.

Documents are organsied into collections (similar to RDBMS tables)

E.g. XML, JSON

There are 4 types of NoSQL database we will be looking at today. The first is the DOD, one of the most common types. Instead of storing the information across multiple tables like a RDBMS, you wrap the data in a document and reference the internal structure to access the data you want. The documents themselves are stored in collections (a collection is like a table where similar documents are stored together). A document can contain any type of data. Instead of using SQL, you would use languages like XML or JSON to retreive the information through an API. To access information from a DOD you just need to reference the document, as opposed to a RDBMS where you have to reference the keys.

JSON

{

    "lastname":"Smith",

  }

A JSON can be thought of as a dictionary of dictionaries (dictionaries are represented by curly brackets). You can see the contact document is wrapped in the bigger collection. XML is a markup language where the data is tagged. You can see the contact document wraps all the information for Mr Smith. The advnatged of a DOD over a RDBMS is that all the information is stored in one place (a document) reducing the need for mapping or joins allowing you to process the data faster.

Key Value Database

The simplest type of NoSQL database, where data (structured or unstructured) is mapped to a key and stored in one location. Data is extracted by referencing the key.

Wrapping several keys in a JSON format creates a document

The simplest type of NoSQL (and a DOD is actually a subset of this) where the information is mapped to a key (like a RDBMS). To retrieve the information, all you need is the key. All the data is stored in one central location (unlike a DOD where it is stored in collections) and any type of data can be stored. Think of Spotify, where all the music will be held in a database and if you want to listen to something you just select the key (which the software will do for you). As these databases do not use SQL, key management is very important and many key value databases incorporate search functions to help you find keys.

Columnar Database

Data is stored in columns instead of rows. SQL can be used to extract data quickly as it will go down the columns for information instead of scanning each row.

A column orientated database applies the row key to each item in a column, allowing it to precisely retrieve information from a select group of columns.

A columnar database stores data in columns, not rows but in many ways function like a RDBMS. They often can be queried using SQL but the difference is that the data can be accessed more precisely as you are only referencing the columns, not scanning through each row. In a row orientated database (like a RDBMS) you assign a row an id (primary key), and when you query, you scan through each row sequentially until you retrieve the information you need. However, this can be inefficient as it will scan each field in each record as well, even when you are applying a filter. A column orientated database applies a key to each item in a row, so when you write a query, it will only scan through the columns you have selected and retrieve the relevant information. This can drastically reduce the amount of time it takes a query to run. Note however, that this process runs each column seperately, so if you are wanting to query multiple columns it may be quicker to use a row orientated database.

Graph Database

Information is stored in nodes with the edges representing the relationships between them.

Each node is a datapoint (e.g. a customer, product, group ,etc) and edges define the relationships (e.g. person (node1) is a member of this group (node2).

Querying is fast as relationships between nodes have already been defined.

A graph database uses graph structures (nodes, edges and properties) to query data. In these types of databases, information is stored in nodes where each node is a datapoint (a customer, product, group, etc- basically a record in a RDBMS). The edges define the relationships between nodes (customer A bought these products, client B is a member of these groups, etc). Relationships are considered top priority, whereas in a RDBMS the relationships are only defined when a JOIN is made. Querying is fast because the relationships have already been defined. An advantage of these types of databases is that data can be interpreted differently depending on how you utilise the edges (retrieve info per product, salesperson, company, date, etc). To retrieve information, you select a node and it will return all the associated information through its pre defined relationships.

Graph Database

A node can contain any type of data (structured or unstructured) including tables

As these types of databases do not use indexing, data retrieval is fast as the query follows the edges to obtain the connected information

Facebook is an example of this where a user (node) is connected to other users and groups by edges

There is no one language for these types of databases as each software will likely use their own. Crucially, a node can hold any type of data, structured on unstructured. Consider Spotify again, each node may be a song which is related to albums, artists and playlists (all of which will be nodes as well). When you want to know all of the songs by a particular artist, you select that node and all the connected songs will be retrieved as well. As there is no need for indexing, there is no need for the software to search and will just pull anything that is already connected making data retrieval faster. Another example is Facebook where each user is a node, connected to other users (friends) and groups they are members of.

Key Features


No Fixed Schema - They do not have to follow historical rules making them more flexible

No Joins - NoSQL databases tend to store data in one large table, taking advantage of cheaper storage and faster processing where redundant data is no longer such a big issue

Size and Scale - No defined limits allows them to scale according to the resources available

The key features of a NoSQL database is that they do not have to conform to strict design structures (a RDBMS has to be a tabular format with keys), they do not require joins as either the data is held in one location or the relationships have been pre defined. This removed redundant data, making storage cheaper and processing faster. And with no limits, there are better opportunities to build larger databases.

'Data Warehousing' is a practice in data management whereby data is copied from various operational systems into a persistant data store in a consistent format to be used for analysis, decision making and reporting.
You may also come across Data Watehouses in your role. These are central data repositories where information from across an organisation are stored (combined using data integration processes like ETL) where stakeholders can make data driven business decisions through the advanced analytics that are on offer. Imagine an organisation where each department has its own DBMS for their own purposes. Higher up decision makers in the company will want to see a unified view of data from across departments (compare sales and inventory for example) so they can make powerful decisions.
Unlike a RDBMS, where data is spread across multiple normalised tables, a data warehouse tends to be one central denormalised table, called the fact table. In there all the information from across the organisation is stored in a consistent format. Surrounding the fact table are dimension tables which define how the data should be aggregated. For example, you may want to analyse the data in terms of customers, or sales, or geographical region. By defining the aggregations you can easily retrieve this information and make decisions based off the analysis.

Key Features

Optimised for low number of complex queries

Contains historical data

Are taken off line when updates are required

Data warehouses are primarily used as a platform for advanced analytics such as recommendation algorithms or generating powerful insights, whereas as RDBMS is more used for storing transactional information for quick references and quick snapshots of the data. As such data warehouses are better suited for advanced queries which derive in depth insights through a deep dive into the data. A reason for this is because warehouses generally do not contain 'live' data (a RDBMS does), but recent data that came with the last upload. This allows for the analytics to run without being interupted because new data is coming in. A consequence of this is that if new data is to be added, the warehouse must come offline for the ETL process to happen. Usually this is done outside of working hours to avoid disruption.

Increased availability of data

Collaboration opportunities

Greater insights and improvements

Benefits

The benefits of using a data wareouse is that more rich data is available to key stakeholders allowing them to make better data driven insights. On top of this, as the data is coming from a variety of sources, the sources themselves have to adhere to a consistent format which means superior quality and consistency for all. Data warehouses allow for collaboration amongst different teams and departments as they are sharing their data and ultimately this will lead to greater insights and improvements in making business decisions. If you have the opportunity to use a data warehouse in your role, you should describe these benefits in your portfolio.
