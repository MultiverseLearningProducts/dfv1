Data Types
You have written your project brief, you have identified stakeholders and agreed deliverables and deadline- but what about the data you are going to use? There is a wide variety of models or visualisations you can choose from when planning a project but to pick the best one requires you to know your data. You need to know how it is structured, what type it is and what permissions exist. This is what we are going to look at today, with first us considering the types of data that exist.

Data can described in one of two ways- quantitative or qualitative. Quantitative data is numerical (think quantities) and can be subdivided into discrete and continuous. Qualitative data is categorical (qualities) and is broken down into binomial, nominal and ordinal.

Quantitative


Discrete

Continuous

Numerical data that can be 'counted'

e.g. number of marbles, siblings, customers, etc

Quantitative data is numerical from which we can calculate aggregates such as mean, median, range and standard deviation. Most visualisations require this type of data (bar charts, box plots, scatter graphs, etc) as we can visualise amounts. The first subset is discrete data, and this is numerical data you can count (or you can think of it as the count of qualitative data) for example, number of marbles in a bag, number of customers in an hour, etc.

Quantitative

Discrete

Continuous

Numerical data that can be 'measured'

e.g. temperature, weight, height

Continuous data can be measured, like temperature, weight or heights. Often you can treat discrete data as continuous when there are too many 'options' (e.g. money). Ask the group if they think age is continuous or discrete.

Categorical data that has two options

e.g. true or false, heads or tails, yes or no

Qualitative

Binomial

Nominal

Ordinal

With qualitative data we are looking at information that describes the quality of something. Some visualisations do exist for this data type (word clouds for example) but often the best way to visualise it is to take the discrete count of each possible category. Binomial data occurs when there are only two options.

Categorical data that has multiple options but no implied order

e.g. colour, job title, error type, etc

Qualitative

Ordinal

Nominal data is categorical with no implied order. For example, car colour is categorical but red is not greater than blue. Most qualitative data will fall under this category.

Categorical data that multiple options and an implied order

e.g. likert scale, coffee cup size, salary band, etc

When nominal data has order then it becomes ordinal. For example, when given a scale from extremely unhappy to extremely happy (likert scale) it becomes ordinal. Ordinal data may be numeric (on a scale from 1 to 5). In this case you have discrete labels (which can be analysed), but the scale itself is qualitative (for example, on a scale from 1 to 5, '3' is a label indicating 'average')

A source is simply somewhere you can extract data for example through a database or through an API.




Sources can typically be broken down into one of four categories- public, proprietary, client and research. Can anyone think of an example of a data source for any of them? We will go through each in more detail.

Public Data

Open Data

Data that can be moved freely, reused and redistributed, although hard to change or modify

Public data is any data that is available to the general public and can be freely used. Typically it will be found on the internet and you can use it in any way you want. Naturally, there are risks with public data, for example security and data cleanliness.

Public Data

Open Data

A subset of public data but:

  • Smaller in volume
  • More likely to be structured
  • More likely to be open licensed
  • Better maintained and more reliable through sanctioned portals
  • May require a nominal fee to be used
A subset of public data is open data. Open data is far more reliable as it will typically be found on approved and regulated repositories (e.g.Kaggle, UK Government, etc). This type of data will be smaller in volume but will be maintained properly and therefore more reliable and safe. Sometimes you may need to pay a fee to the repository to access their data (monthly subscriptions, etc) but the data itself can still be used as you would like.
According to the Open Knowledge Foundation:
“Open data and content can be freely used, modified, and shared by anyone and for any purpose.”
Even though open data is more reliable than public data, it can still be used by you for whatever purpose you would like.
Public data can be defined as all information in the public domain, encompassing anything from a monthly updating dataset on a government data portal to PDF files that are only accessible via Freedom of Information requests (and everything in between).

Administrative

Data that is owned and stored within an organisation. Proprietary data may be protected by patents, copyrights/trademarks or trade laws.

Proprietary data is data owned by an organisation, so much of the data you will use in your role. There are much stricter rules around proprietary data as often it will contain sensitive information about the organisation and/or its customers/clients. When using this sort of data you need to adhere to the rules set and only use it for the reason you said you would.

Data Protection Act (DPA 1998)

1 Data must be kept secure
2 Data stored must be relevant
3 Data stored must be kept no longer than necessary
4 Data stored must be kept accurate and up to date
5 Data must be obtained and processed lawfully
6 Data must be processed within the data subject rights
7 Data must be obtained and specified for lawful purposes
8 Data must not be transferred to countries without adequate data protection laws
Speaking of legal issues, we also need to consider the law when it comes to data. The data protection act (1998) was set out by the UK government to ensure all data is processed and used lawfully. For your exam you will need to be aware of what is in the DPA.

General Data Protection Regulation
(GDPR 2018)

1 Data must be processed lawfully, fairly and transparently
2 Data must be collected for specified, explicit and legitimate purposes
3 Data must be adequate, relevant and limited to what is necessary for processing
4 Data must be accurate and kept up to date
5 Data must be kept only for as long as is necessary for processing
6 Data must be processed in a manner that ensures its security
In 2018 the General Data Protection Regulation was introduced for all companies that operate in the European Economic Area. This supersedes the DPA and it is important in your portfolio that you demonstrate that you are aware of it and have complied. Even though the UK has left the EU, most organisations still work with EU based clients and therefore still have to adhere. The penalities for non compliance can be severe with harsh fines, loss of customer confidence and poor market reputation if any breaches occur. GDPR (and the DPA) exist to protect personal data (such as name, contact information, IP address, personal information, political preferences and more). It is esnetial then if you are working with this sort of data that you demonstrate compliance and explicitly state what you have done to protect it.

A quick note, data may look structured but isn't. For example, an Excel spreadsheet is unstructured . The data within may be in a structured format which you can analyse, but the sheet itself isn't. This is because Excel doesn't have to conform to structured rules- you don't need column names, columns don't need to have the same amount of information, you can have multiple tables in a single sheet. So while the data within the spreadsheet can be structured, the sheet itself isn't.

Finally a tree, where data is defined by its relationship to each other. An exmaple of this is folders on your computer.

This process is how data is collected from a variety of disparate sources and combined into one unified target view. Consider your organisation, it will likely be split into several teams and divisions, each collecting and storing data relevant to their roles. Sales teams will hold data on customers while the inventory team know how much they have left in stock. For key decision makers in the company, they will want all of this data in one central location so they can get a quick view to understand the situation and make data driven decisions. This is what ETL is for, combing data from across an organisation into one central place (often a data warehouse).

Extraction is the process of gathering data from a variety of disparate sources

The extracted data is usually copied from the source, not moved

Validation occurs at this stage to ensure the data is in the correct structure and format, as well as ensuring necessary permissions have been given

The process can be continuous or done in batches

Extract

Transform

Load

The first stage in the process is extraction. This is when you identify and collect the data from all the sources you require. Note, you don't 'take' the data but a copy is made and sent to the warehouse. This means the original source is left untouched. It also means that if any changes are made to the source, these will not be reflected in the warehouse until an update is made. A consequence of this is that warehouses are static (making querying and processing quicker) but potentially out of date if the source changes (fast flowing databases like sales or inventory will change throughout the day, so ETL updates may need to happen regularly). Often when a warehouse is updated it will need to be taken 'offline' so extraction tends to happen outside of working hours to avoid disruption for users. You will validate your sources at this time to ensure they contain the correct information.

Transformation is the process of ensuring the extracted data is in a consistent format

This can include removing null values, changing data types and ensuring field names are the same

As the extracted data is a copy, the original will remian unchanged

Extract

Transform

Load

Once the data has been pulled to the warehouse it is time to transform it. This means all the data needs to be in the same consistent format- data types need to be the same, null values should be removed and column names should be consistent (especially if they appear in more than one source- more on this in our next module). Note, as the data is copied from the source, the original data is unchanged by this process.

Loading is the process of joining the transformed data together into a single unified view (called the target)

Data verification is undertaken post loading to ensure the combined data is accurate and fulfils the necessary business requirements

With 'Big Data' this process is done using parallel processing to manage the large volume of data being written to the system

Finally, once the data is in a consistent format it is time to join it all together into one table. At this point you will verify that the data stored is correct and useful for the purposes you require. A few things to note, with large voluminous (and potentially fast flowing) data this process may be more cost effective to be run using a Big Data technology which uses a form of parallel processing to carry this out. As mentioned earlier, data warehouses are static and can quickly be out of date with the the original sources. It is highly likely then that an ETL process will be carried out regularly to ensure decision makers always have access to the latest data.

Information Structure and Rules


Data integration activities for data warehouses requires that you follow some basic rules:

  • Security policies must be specified by organisations providing data sources to prevent data leakage and unauthorised access
  • Access layers (e.g. networks, firewalls, servers, etc) between sources and targets should be properly configured (especially of data is sourced externally)
  • Integrated data should be immutable- you should not be able to change the data once it is stored in the unified view
  • Validation checks should be carried out during ETL:
    • Source and target table structures and data types should be consistent
    • Column names should be the same as defined by a mapping document
With ETL, it is important you follow certain rules to ensure the process runs smoothly and safely- you don't want incorrect data being fed to key decision makers...

Information Structure and Rules


Data integration activities for data warehouses requires that you follow some basic rules:

  • Verification is also carried out on the target:
    • Verify that the data is accurate
    • Verify the data is the 'right' data to be stored in the target
    • Verify the data has not been duplicated
A possible project for your portfolio could be to build a data warehouse or pipeline for someone in your business. If this is the case, make sure you capture the business requirements clearly before beginning. This way you can guide the process precisely and be able to verify at the end if you have met the requirements of the stakeholder.
