Principles of Data Classification
The focus on the previous session was identifying and managing stakeholders to ensure success in our projects. However, we alo need to consider the data we are going to use. The process of categorising our data so it can be efficiently and effectively protected is called data classification. It is an important and necessary step in the data analytics life cycle due to issues relating to compliance and regulation. You must demonstrate this process in your portfolio.

Things to consider

What are the data types?

Where is the data stored?

Where did the data come from?

Is any data sensitive?

Who has access to the data?

How is the data protected?

How are you going to comply with GDPR?

What should we consider as part of the data classification process? We should consider data types and storage as this will help us identify how the data can be used. We should think about its source as that will determine what we can do with it and how much we can trust it. We also need to consider if we are handling any sensitive data, how we are protecting it and controlling access. The rest of the session will help us answer these questions. Do you think there is anything else that should be included? In your portfolio you will need to demonstrate a reasoned application of data classification, including describing any flexibilities you have applied and its purpose (e.g. how your handling of PII complied with GDPR).
Content Context User
Anything to do with data classification can be summarised into these three areas:
  • Content- what is in our data? This includes checking for sensitive data such as PII, key performance indicators and other information that company would rather not be made publically available.
  • Context- Check the application, location and owners of the data. How is it being used? Who has access? When are the accessing it?
  • USer- What do you know about the data? Using your knowledge and discretion, go through and flag anything that could be sensitive.
Data Types
The first part of data classification is looking at the data itself. Data can come in a variety of formats and could contain many different things. Let's look at the different types of data we can encounter. Data can described in one of two ways- quantitative or qualitative. Quantitative data is numerical (think quantities) and can be subdivided into discrete and continuous. Qualitative data is categorical (qualities) and is broken down into binomial, nominal and ordinal.

Quantitative


Discrete

Continuous

Numerical data that can be 'counted'

e.g. number of marbles, siblings, customers, etc

Quantitative data is numerical from which we can calculate aggregates such as mean, median, range and standard deviation. Most visualisations require this type of data (bar charts, box plots, scatter graphs, etc) as we can visualise amounts. The first subset is discrete data, and this is numerical data you can count (or you can think of it as the count of qualitative data) for example, number of marbles in a bag, number of customers in an hour, etc.

Quantitative

Discrete

Continuous

Numerical data that can be 'measured'

e.g. temperature, weight, height

Continuous data can be measured, like temperature, weight or heights. Often you can treat discrete data as continuous when there are too many 'options' (e.g. money). Ask the group if they think age is continuous or discrete.

Categorical data that has two options

e.g. true or false, heads or tails, yes or no

Qualitative

Binomial

Nominal

Ordinal

With qualitative data we are looking at information that describes the quality of something. Some visualisations do exist for this data type (word clouds for example) but often the best way to visualise it is to take the discrete count of each possible category. Binomial data occurs when there are only two options.

Categorical data that has multiple options but no implied order

e.g. colour, job title, error type, etc

Qualitative

Binomial

Nominal

Ordinal

Nominal data is categorical with no implied order. For example, car colour is categorical but red is not greater than blue. Most qualitative data will fall under this category.

Categorical data that multiple options and an implied order

e.g. likert scale, coffee cup size, salary band, etc

Qualitative

Binomial

Nominal

Ordinal

When nominal data has order then it becomes ordinal. For example, when given a scale from extremely unhappy to extremely happy (likert scale) it becomes ordinal. Ordinal data may be numeric (on a scale from 1 to 5). In this case you have discrete labels (which can be analysed), but the scale itself is qualitative (for example, on a scale from 1 to 5, '3' is a label indicating 'average')

Sometimes data looks structured but isn't. For example, Excel spreadsheets have no rules around usage, so you can have multiple tables or different data types in one column.
A quick note, data may look structured but isn't. For example, an Excel spreadsheet is unstructured . The data within may be in a structured format which you can analyse, but the sheet itself isn't. So while the data within the spreadsheet can be structured, the sheet itself isn't.

We tend to think of data as being in a row and column format, but that is not the only way it can be given to us. For example, data often comes in files (can anyone think of an example? CSV, XLSX, JSON, etc). The data contained within the file itself can be structured or unstructured depending on what type it is, in a few minutes we will look into file types in more detail. A list is a a 1-dimensional (think a single row) object where all the data is stored and is the basic building block of structured data. When we start to add more dimensions (rows and columns for example) the data becomes an array where we locate the data using indices (think coordinates). Most of the data we use is 2-dimensional (row and column) but it is possible to have more dimensions. A 2-dimensional array becomes a table (the one you will be most familiar with) when the rows and columns are labelled (records and fields), this is your typical structured data. Finally a tree, where data is defined by its relationship to each other. An exmaple of this is folders on your computer.

A data source is the location from where we collect it from. Sources can typically be broken down into one of four categories- public, proprietary, client and research. Can anyone think of an example of a data source for any of them? We will go through each in more detail.

Public Data

Open Data

Data that can be moved freely, reused and redistributed, although hard to change or modify

Public data is any data that is available to the general public and can be freely used. Typically it will be found on the internet and you can use it in any way you want. Naturally, there are risks with public data, for example security and data cleanliness.

Public Data

Open Data

A subset of public data but:

  • Smaller in volume
  • More likely to be structured
  • More likely to be open licensed
  • Better maintained and more reliable through sanctioned portals
  • May require a nominal fee to be used
A subset of public data is open data. Open data is far more reliable as it will typically be found on approved and regulated repositories (e.g.Kaggle, UK Government, etc). This type of data will be smaller in volume but will be maintained properly and therefore more reliable and safe. Sometimes you may need to pay a fee to the repository to access their data (monthly subscriptions, etc) but the data itself can still be used as you would like.
According to the Open Knowledge Foundation:
“Open data and content can be freely used, modified, and shared by anyone and for any purpose.”
Even though open data is more reliable than public data, it can still be used by you for whatever purpose you would like.
Public data can be defined as all information in the public domain, encompassing anything from a monthly updating dataset on a government data portal to PDF files that are only accessible via Freedom of Information requests (and everything in between).

Data Quality
We have discussed what our data might look like, where it is stored and how we can access it. The next part of data classification is checking its quality- how useful, clean and truthful it is. Can anyone think of factors that might lower our quality of data?
Accuracy Complete Consistency Uniqueness Timeliness
When talking about data quality we think of these five areas:
  • Accuracy- how accurate (correct) is your data? Are names spelled correctly? Are values accurate?
  • Completeness- Is all the data present? Are there missing values? Do the field names make sense?
  • Consistency- Is there a consistent format throughout? Including, currency, data types, units, etc. Are there any outliers?
  • Uniqueness- Is each record unique? Are there any duplicate values? Are real world entities represented more than once (e.g. apple and Apple)?
  • Timeliness- Is the data up to date? Your data should be current with respect to the needs of the business

When writing up your portfolio you must demonstrate and discuss the steps you took to inspect and if necessary, clean the data to ensure it was of the highest quality for your analysis. We will cover practical examples of this in our technical bootcamps.

If you find an error...

You should either...

→ Correct it

→ Impute a new value

→ Remove it

→ Ignore it

Often the data we recieve will not be clean, and there could be errors. What sort of errors might we come across? Missing values, corrupted data, outliers, incorrect values, wrong data types, etc. Generally when we come across a problem there three things we can do. We can correct the error, finding the correct value or fixing the mistake. We could impute a new value (replace the erroneous data with 0, or with the mean/median value for that field). If we cannot correct the mistake, or find a suitable value to replace it, you could remove the data point but this should only be done with caution as this could be difficult to undo. Another option would be to ignore the problem if you feel it won't affect your work, or if the error means something which should be investigated
Whatever the issue, you must ensure that a solution for the true root cause is identified to prevent recurrence
Whatever the issue is, whether it is unclean data, or a failure in the database itself, you must demonstrate that you have found a solution to the true root cause. For example, if the data is corrupted when you receive it, it is possible there is something wrong in the database. You should contact the database administrator so they can fix it. If you are collecting the data yourself and feeding it into a pipeline, you need to put into place checks to ensure high data quality and put in safeguards to protect corruption or erroneous data.

Throughout this session we have beek skirting around a piece of legislsation that guides us on how to use data...

General Data Protection Regulation
(GDPR 2018)

1 Data must be processed lawfully, fairly and transparently
2 Data must be collected for specified, explicit and legitimate purposes
3 Data must be adequate, relevant and limited to what is necessary for processing
4 Data must be accurate and kept up to date
5 Data must be kept only for as long as is necessary for processing
6 Data must be processed in a manner that ensures its security
In 2018 the General Data Protection Regulation was introduced for all companies that operate in the European Economic Area. It is important in your portfolio that you demonstrate that you are aware of it and how you have complied. Even though the UK has left the EU, most organisations still work with EU based clients and therefore still have to adhere. The penalities for non compliance can be severe with harsh fines, loss of customer confidence and poor market reputation if any breaches occur. GDPR (and the DPA) exist to protect personal data (such as name, contact information, IP address, personal information, political preferences and more). It is esnetial then if you are working with this sort of data that you demonstrate compliance and explicitly state what you have done to protect it. What About Consent?


Whenever you subscribe to a new service you will be asked to consent for your data to be used in a particular way. If as part of your work are collecting information directly from clients/customers then you will need to demonstrate an example of a consent form. Keep in mind that the form will need to inform the person what the data will be used for, where it will be stored, for how long, who else will have access, how they can access it, amend it and delete it to be GDPR compliant.

Consent is“any freely given, specific, informed and unambiguous indication of the data subject’s wishes by which he or she, by a statement or by a clear affirmative action, signifies agreement to the processing of personal data relating to him or her”.

Article 4(11) GDPR

Article 7 of GDPR also states that you must keep records of consent. It cannot be stressed enough that PII must be given consenually and only under circumstances where the data owener understands how and why their data will be used.

While GDPR states data should only be kept as long as necessary, it does not define an actual time limit. What you must rely on then, is your company's definition of reasonable use. If the data you are using does not meet the criteria it must be deleted, which includes all digital, hard and backup copies of the data. You cannot hold onto someone's data 'just in case'. When might data no longer be needed? E.g. client/customer has moved on, project has been completed, etc. If in one of your projects you have used PII, you will need to cite your companies policy on reasonable use and state the conditions upon which you will delete the data.

How do we keep data secure then? The simplest way is for companies to have defined policies and ensure all employees follow it through training. Data should be stored securely with special access permissions granted to those who have justified they need it. This access should regularly be reviewed, and where necessary have access revoked to those who no longer need it. The data should be processed on company machines or a secure VPN. If you are carrying hardware (laptop, USB) with PII you must ensure you are keeping it secure- the safest way is to leave it at work. As more data is being stored remotely it is becoming less likely that you will every 'carry' sensitive data, but if you do you must ensure it is being kept safe. You should document in your portfolio how you complied with your company's policy on data security.

Contributing to a Safe Environment

We will consider this more during the technical bootcamps, but whenever we are using data we need to consider the impact it can have on our lives- both personally and professionally. It is all too easy to build a model that is unintentionally biased or discriminatory due to the quality of the data we are using. Some PII is considered 'special categories' and includes race and ethnicity, gender, religious beliefs and political opinions. If you are using such data, it is your responsibility to ensure it is treated in such a way that does not lead to a biased outcome. The easiest way is to mask the data in such a way that each person can not be identified and so that it is then treated equally. It is the responsibility of all of us to contribute to a safe and secure working environment through the way we handle data. In your portfolio, where appropriate, you should discuss what you have done to promote the ethical use of data.

Everybody who uses personal data in their role must comply with GDPR

If you are using PII then you must comply with GDPR, it ensures the rights of the people who own the data are protected. Data breaches often come with severe punishments and a loss of market reputation, so you must do your part to ensure the data is protected.
