Data Synchronization
Data synchronization is a process similar to data integration that apprentices need to be aware of for their KM1 exam, in this session we will be covering the basics to understand what it is and how it can be implemented in our roles.
What is Data Synchronization?
Ask apprentices to annotate the screen with their ideas for what data synchronization is.
Data synchronization is the process of keeping two or more systems in sync so that any updates in one are automatically reflected in the other to ensure consistency.
Read the quote and ask apprentices if any use such processes in their role. Essentially data synchronization is about instantaneous updates so all users are using the most up to date versions and information at any one time. Unlike data integration which typically requires batch uploads and downtime, a data synchronization process should have 100% uptime with continuous flow of data and updates.
Benefits of Data Synchronization
Now we know what data synchronization is, lets understand why it is useful for our workplaces.

Benefits

Increased accessibility and availability of data

Increased security and regulatory compliance

Avoids data conflicts

More efficient decision making from using up-to-date data

Data synchronization allows for instant updates in a file meaning any user is going to have access to the most up to date version which will allow for more efficient decision making. It is also easier to avoid conflicts as problems can be identified in real time and therefore dealt with (unlike data integration where conflicts may arise if the sources were not scoped properly). Ask apprentices if they can think up any potential issues around synchronization (e.g. may be hard to maintain control, cost of keeping something online 100%, velocity of systems with large amounts of data, etc)- we will cover these later.
Methods of Data Synchronization
Let's now look at the various ways a data synchronization system could be put to work.

File Synchronization

Version Control

Distributed File System

Mirror Computing

Methods

There are four main methods that we will look at in more detail.

File Synchronization

The process of ensuring that files in two or more locations are updated in real time via certain rules. It should occur automatically and prevent duplication of data.

File synchronization ensures that when one file is updated in one location, all other locations are updated too. It is like instant messaging on WhatsApss (or other similar products) where when one person sends a message, all other people in the thread see what it is instantly. Ask apprentices if they can think of other examples of this.

Features

Can be done locally or in the Cloud

Updates your files in real time

Two-way synchronization

Allows for collaboration between developers

Detects conflicts

Go through the features, this method of synchronization is ideal for instant collaboration as when one edit is made the other user can see what it is.
Examples of this include OneDrive, Google Drive, Dropbox, box and GoodSync. Ask apprentices if they use any of these and what the benefits and challenges are.

Version Control

Synchronization occurs over multiple devices, allowing users to simultaneously edit their own version. Updates are reflected when the versions are pulled together.

In file synchronization any updates to a file are reflected immediately for all users, as all users are accessing the same file from different locations. In version control users will be using (and updating) different versions of the same file such that if an update is made, no-one else is made aware until the different versions are merged together. In other words, changes are not updated automatically.

Features

Can be version or revision control

Allows movement between different revisions

All changes are tracked

Allows for different team members to develop different aspects of the project

This process allows for different people to work on the same project at the same time without the risk of their work accidentally being overwritten or messed around with as they are working. All changes are tracked (every user can review these) so if a mistake is made, it is easy to revert to a previous version. When merging versions together it is possible conflicts will arise (two people have edited the same part) so there will need to be a conflict resolution process to deal with these.
Common software includes GitHub, Git and GitLab. Ask apprentices if any use these sort of processes and what difference it makes to their role.

File synchronization allows for simultaneous working on a project

i.e. any updates are reflected instantly for all users

Version control allows developers to work independently on a project and then update each other's versions when ready

Most version control tools integrate some form of file synchronization

So apprentices are clear on the difference between the two systems: file synchronization allows for instant updates so is more agile, version control allows users to work independently so is easier to control and spot conflicts/mistakes. Both have their advantages (and disadvantages) but in truth, both incorporate something of each other. For example, Google suite applications such as Google Documents allow for file synchrnonization while also maintaining a version control to see all updates. Most version control tools use some sort of file synchronization such that when an update is made, all users can see the change before merging.

Distributed File System

Allows users in different locations to access the same files and systems through a common file system, for example over a local area network (LAN).

A distributed file system (DFS) is very common where users are accessing shared files over a Local Area Network (LAN). For example, having a shared drive on your PC. Where this differs from file synchronization is that it often doesn't allow multiple users to edit the same file at the same time- other users would have to wait until the person editing has saved and closed the file.

Mirror Computing

A mirror is an exact copy of a dataset. It is used to provide different sources with an exact copy of a dataset. More useful for backing up data.

In mirrored computing the file/dataset is copied and sent out to the different users, but there is no intention of merging these files back together again later (unlike version control). This method is ideal for creating backups.

Challenges of Data Synchronization
Now we have looked at different methods of data synchronization and considered a bit of their uses, we should also look at some of the challenges and how we can mitigate them.

Challenges

Security

Management

Data Quality

Data Format

Performance

Data Ownership

Ask the apprentices what they think these different challenges might refer to. We will cover each in more detail over the following slides.

Security

The Issue

Data breaches and data leaks

Ensuring data sharing conforms to regulatory standards

Could lead to a loss of reputation and/or revenue

An issue with data synchronization is around security- ensuring that data is not leaked which could lead to issues with GDPR and other consequences. This would normally arise due to the wrong people been given access to the system. Ask the apprentices to suggest solutions.

Security

The Solution

Put in place data policies and access requirements

All synchronization must follow security needs

Implement data encryption

Simply, the solution is to have policies and access requirements- including a named person (or persons) whose job it is to manage access. This will include reviewing regularly who has access and revoking it if no longer required. It will also help to implement some sort of encryption process.

Management

The Issue

Inconsistencies in synchronization system updates

Difficult to catch errors during synchronization activities

This is to do with the management of the system, ensuring the process works such that if an update is made it is available to all users. Errors can arise due to problems with the system itself or within the data (corruption). These problems can be difficult to catch until the synchronization is complete and users start complaining.

Management

The Solution

Data Management systems put in place for managing and integrating data in real time

Ensure the systems and processes work by troubleshooting and putting in place regular maintenance.

Data Quality

The Issue

Data integrity is easily compromised due to access from multiple devices

If everyone has editing access to the data, how do you ensure that errors are caught? For example, someone well-meaning could make an edit which is wrong, or worse, corrupts what is already there. Or, if using Version Control, what if someone makes a useful update- how do you ensure other users merge their version to receive it? Any ideas?

Data Quality

The Solution

Implement constant data validation strategies into synchronization tools across all sources

Again it comes down to policies and systems. Ideally there should be someone who is in charge of version control and maintains the 'master' version to ensure it is always up to date and correct. You may also want to implement a system where if someone wants to make an edit, it needs to be approved by a moderator before going live. Software like GitHub alerts the repository owner if there are any conflicts with versions which can help manage data quality.

Data Format

The Issue

Different data formats lead to complexity issues

An issue can occur if within the system there are different data structures, so what works for one file may not work for others. This can cause frustration around setting the process up or maintaining it if errors start to arise.

Data Format

The Solution

Ensure data format compliance is consistent across organisation by having policies and templates

A simple solution to this is to ensure all data formats are consistent (again having a moderator for this is useful). Having templates available for new users will be useful as well.

Performance

The Issue

Lag time during peak useage of if one of the synchronization steps fails

If there is a lot of data being synchronized at once, or a lot of users then that can lead to performance issues (very frustrating if trying to collaborate).

Performance

The Solution

Ensure proper capacity planning is put in place and synchronization activities made a priority

Make sure the system has the capacity to handle the amount of data flow and access required. If needed put restrictions on how many peeople can access it at any one time.

Data Ownership

The Issue

Difficult to track changes if anyone in an organisation has access to the file/datasets

If too many people have access, edits can be made without others knowing. This includes potentially junior members of an organisation overwriting the work of senior members.

Data Ownership

The Solution

Restrict access to only those who need it and have a referral process so others can suggest changes

As stated earlier, control access to the system only to those who need it and review this regularly. For import/core files have edits be approved by a moderator.

What is the format?
In your synoptic project there are 6 stages you will follow and write up on. We will go through each of those stages in detail. Keep in mind, this is not quite the same as the STAR format of the portfolio pieces.

Data Discovery


A short review and document of key fields to be used for analysis

First is data discovery where you will write a short review of the data you used for your analysis in the project.

Data Discovery


  • Summarise the question asked
  • State the metrics you are using and which column names (in the data) they correspond to
  • Explain why you have chosen these fields
  • State your assumptions
Some advice what to write about- refer back to your project brief to justify which fields you selected and what metrics you used to evaluate them. You will be expected to have assumptions, so state them!

Data Preparation


A short summary of the processes used to extract, transform and load the data

Next you will write about what you did to prepare your data for analysis (basically where you will evidence your EDA)

Data Preparation


  • This is your exploratory data analysis section
  • What trends did you see?
  • Document any issues with data quality
  • What did you do to resolve these? Why did you choose those methods?
  • What methods were best for your purpose? How did you decide?
Talk about your data cleaning (what, why and how) and what visualisations you chose to demonstrate it. Did you see any trends or patterns that might impact your analysis? What else might you want to consider?

Model Planning


Document the final selected models, charts and reports

Then you will write about what model you planned to use.

Model Planning


  • What model did you intend to use for your investigation? What features did you plan to select?
  • Why did you come to this decision?
  • What visualisations did you intend to use to demonstrate this?
  • Why did you choose them?
This is also part of your EDA. Based off your analysis what type of model (regression/time-series/classification etc) did you think would be appropriate to use, how did you come to this conclusion? What visualisations did you produce to justify this? Keep in mind this is part of planning, you are not yet talking about how you built the model- just what you planned to do.

Building


Build the reports/models to answer the questions

In this section you will discuss the model you built.

Building


  • Do the analysis
  • Train and test the model
  • Pick the best version of your model
  • Visualise the results
  • Have you answered all of the questions with these?
  • Can you show off any relevant additional skills?
You will describe the models you built. Did you use train test split? Did you build several models? If so, which one did you chose to progress with? Why (metrics)? How did you visualise the result? Link ths back to the project brief to show you are answering the question.

Testing


Document how you have tested and validated the results

Next you move onto describing how you validated the result.

Testing


  • Are your results significant? What statistical tests have you performed to show this?
  • Have the assumptions you made about the data invalidate your conclusions?
  • How do you know your results are valid?
How do you know your results are significant? You should demonstrate the statistical tests you used and again go over the assumptions you made and whether they invalidated your conclusion (how did you check these assumptions?)

Communication


Compile a short report answering the questions asked in the Project Objectives section

The final section will link back to the project brief and the question you were trying to answer.

Communication


  • Answer the questions
  • Include screenshots of your visualisations where appropriate
  • Add in any additional analysis you would like considered
  • Check you answered the questions (again...thoroughly)
Go back to the question you were trying to answer and summarise how your work has addressed it. Or if it hasn't, why it didn't and what you would do differently if you could do it again.

Your project is your chance to shine, use it show off your strengths or cover a skill that your portfolio may have missed (like regression)

Interview

What is it?

An opportunity for the Assessor to ask you for further details, which has three main uses:

Allows you to show off

Confirms the skills you've demonstrated in your portfolio

Demonstrate a 'weaker' standard

Again the interview is a chance for you to show off all the skills you have learned. Go in prepared to talk about how you have used data analytics skills to enhance your role (whether you are an analyst or not).

Welcome and introductions

Photo Id check

Overview of purpose and structure

You will then be asked to explain your job role, type of tasks in a typical week and the type of environment

The Structure

After you have confirmed your identity the rest of the interview will be a conversation about how you have applied data analysis skills to your role, referring to your portfolio, employer reference and synoptic project as evidence.

Structured Discussion


Explore what you have done and how you've done it

  • Portfolio, synoptic project and employer reference form
  • Specific comments documented


Explore and identify the range and depth of knowledge, skills and behaviours

Confirm and validate the quality of your work

Opportunity to provide further evidence (what else have you been working on?)

Ask the apprentices if they have any questions.

How should you answer?


Use the STAR format, but sometimes it may not be necessary to use all four sections.

Use the assessor's question to determine the type of answer expected

  • Why was it important to perform this analysis?
  • How did you ensure your analysis was correct?
  • How did you ensure your insights were valid?
  • What was the impact of...?
Keep in mind the STAR format- what was the situation (context), proposed solution, actions you carried out and the result. Always remember to include who you reported the results to, the impact it had and why it was important.
